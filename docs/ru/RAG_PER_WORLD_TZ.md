# ТЗ: Пер-мировый RAG (гигиена кэшей, шаринг миров, слот под Ladybug-индекс)

Статус: Фаза А — реализуется; Фаза Б — отложена до появления производителя (bake);
Фаза В — трек редизайна RAG (Ladybug, двухтиерный стор).

## 1. Контекст и инварианты (проверено по коду)

- Персистентного индекса нет: документы ретрива пересобираются на каждый запрос из
  живого `World` (`retrieval_documents` — память + fact records + сцена + ростер +
  whereabouts; `memory_documents_for_access` — память). BM25 считается в памяти.
- Единственный персистентный артефакт — кэш эмбеддингов
  (`embeddings(model, text_hash, text, dims, vector_b64, created_at)`,
  PK `(model, text_hash)`), где `model = "{GM_RAG_EMBEDDINGS_MODEL}@{GM_EMBEDDER_QUANT}"`,
  `text_hash = sha256(py_strip(text))`. Кэш — чистый warm-start: промах пересчитывается
  сайдкаром. Корректность шаринга уже гарантирована ключом: чужая модель → промах,
  никогда не «битые» вектора.
- Кэш хранит ПОЛНЫЙ ТЕКСТ документов → любой файл кэша, попавший в пакет мира,
  уезжает получателю вместе с текстами (экспорт — слепой рекурсивный zip каталога
  пакета, исключаются только `.*.tmp`).
- Worldgen сеется НА ЗАПУСКЕ (`WorldSpec{seed: new_dice_seed()}`): у получателя тексты
  документов процедурного мира другие → «запечённые» вектора почти не попадают в кэш
  по content-hash. Стабильные чанки появятся только в Ladybug-tier-1.
- Движок был процесс-глобальным синглтоном с ОДНИМ путём кэша из первого конфига.

## 2. Фаза А (СЕЙЧАС): изоляция кэшей по мирам + GC + защита экспорта

### 2.1 Хранение
- Пер-мировый rw-кэш: `<GM_RAG_WORLDS_DIR>/<world_id>.sqlite3`.
  `GM_RAG_WORLDS_DIR` — новый env/поле конфига, дефолт `<data_dir>/rag_worlds`
  (то же разрешение, что у `GM_RAG_CACHE_PATH`; НЕ выводится из родителя пути кэша,
  НЕ живёт под `library/`). Имя файла — `world_ref.id`, санитизированный до
  `[A-Za-z0-9_-]`; иначе детерминированный фолбэк от sha256(id).
- Глобальный `gm_lab_embeddings.sqlite3` — без изменений; остаётся кэшем для сессий
  с `world.world_ref == None` (built-in истории) .
- ЖЁСТКОЕ ПРАВИЛО: в Фазе А рантайм НИЧЕГО RAG-ного не пишет внутрь
  `library/worlds/<id>/`. Инвариант несёт приватность экспорта.
- Миграции старых строк глобального кэша нет: мир-привязанные сессии лениво
  переэмбеддят в свой файл (самолечится, единоразовая цена).

### 2.2 Маршрутизация движка (ВСЕ ТРИ точки входа)
- Точки входа, маршрутизируемые по `world.world_ref`: fact-путь
  (`build_retriever`/`retrieve_world_fact_report`), memory-путь
  (`retrieve_memory_rows_report` → `..._with_documents`) и обёртка
  `retrieve_memory_rows`. Память — тоже тексты сессий этого мира, изоляция
  обязана покрывать её (иначе дыра в гарантии).
- Вместо синглтона — процесс-реестр движков, ключ = разрешённый путь кэша
  (глобальный путь — сентинел для `world_ref == None`). Семантика «первый конфиг
  выигрывает» сохраняется, но теперь пер-ключ. Один `init_db` на мир на процесс —
  ничего не строится на горячем пути заново.
- Контракт конкурентности (задокументирован в коде): две сессии одного мира делят
  один файл; WAL + busy_timeout=10000 + content-addressed `INSERT OR REPLACE`
  делают конкурентных писателей безопасными.

### 2.3 GC
- Удаление мира: best-effort `delete_world_cache(world_id)` (файл + `-wal`/`-shm`),
  на ОБОИХ серверных сайтах удаления. Нефатально, как существующий purge-хук.
- Импорт мира с `overwrite=1`: тот же GC для переиспользованного id — повторный
  импорт под тем же id никогда не сервит кэш предыдущего мира (отравление при
  reuse id).

### 2.4 Скоуп-пёрдж
- `purge_embeddings_for_texts` получает скоуп мира: сайт удаления чата в
  gml-persistence знает `world_ref` сессии из payload → пёрджится ТОЛЬКО файл
  этого мира (или глобальный для `None`). Никаких глоб-свипов по всем мирам.

### 2.5 Экспортные предохранители (share.rs)
- Фильтр zip-обхода дополнительно пропускает `*-wal`, `*-shm`, `*-journal`
  (sqlite-сайдкары) по всему дереву. Сам `rag.sqlite3` НЕ фильтруется — это
  будущий пакетный слой Фазы Б, он ДОЛЖЕН уезжать.
- Тесты: сайдкары не попадают в архив; рантайм-ретрив мир-привязанной сессии
  создаёт файлы только под `GM_RAG_WORLDS_DIR` и не трогает каталог пакета.

## 3. Фаза Б (ОТЛОЖЕНА: нужен производитель — bake). Спека зафиксирована, чтобы её нельзя было построить неправильно

Пакетный warm-start слой `library/worlds/<id>/rag.sqlite3`, read-only на рантайме,
пишется ТОЛЬКО явным bake-шагом. Требования (закреплены адверсариальной панелью):
1. Bake пишет базу в `journal_mode=DELETE` (или OFF) — НИКОГДА не шипить WAL:
   читатели создают `-wal`/`-shm` прямо в пакете, а RO-подключение не может
   сменить journal mode.
2. Рантайм-открытие: `SQLITE_OPEN_READ_ONLY` + PRAGMA `query_only=ON`,
   `trusted_schema=OFF`, `cell_size_check=ON`, `mmap_size=0`, малый busy_timeout;
   НИКОГДА не выполнять WAL-прагму на этом подключении (нужен отдельный RO-режим
   `EmbeddingCache`, не `connect()`).
3. Файл приходит из НЕДОВЕРЕННОГО zip: `PRAGMA quick_check` при первом открытии;
   валидация схемы (в базе — ТАБЛИЦА `embeddings` с ожидаемыми колонками, без
   view/триггеров); бюджет на объём чтения; ЛЮБОЙ сбой → слой молча пропускается
   (это только warm-start).
4. Штамп — ВНУТРИ базы: таблица `meta(key,value)` c `format=gmlab.rag-cache/1`,
   `embedder=<model@quant>`, `dims`, `docs` (+ зарезервирован `reranker`).
   НЕ в world.json: `WorldEnvelope::to_file_value` пересобирает манифест из
   фиксированного набора ключей и молча теряет незнакомые при следующем сохранении.
5. Bake имеет смысл только когда тексты документов стабильны между машинами —
   т.е. после Ladybug-tier-1 (детерминированные чанки библии мира). Для
   сегодняшних процедурных миров hit-rate ≈ 0 — потому и отложено.
6. При появлении bake — аудит пути `zip_story_with_world` (запечённый мир
   бандлится слепо).

## 4. Фаза В (трек редизайна RAG — Ladybug)

Двухтиерный стор: tier-1 — детерминированные чанки библии мира + векторный индекс
в пакете как derived-артефакт (тот же слот и требования, что Фаза Б); tier-2 —
сессионная память локально (слой Фазы А). Cross-encoder rerank — по выводам бенчей.
Фаза А — ровно тот seam (RO-пакетный слой + rw-локальный слой), в который это
втыкается без переделки.

## 5. Отвергнутые варианты (чтобы не возвращаться)

- Rw-кэш рантайма ВНУТРИ каталога пакета + фильтр на экспорте: privacy-footgun —
  один пропущенный кейс фильтра шипит тексты сессий игрока.
- Вывод `rag_worlds_dir` из родителя `GM_RAG_CACHE_PATH`: ломает герметичность
  (тесты ставят путь кэша в системный `%TEMP%`); гигиену нельзя вешать на родителя
  файлового пути.
- `Config.packages_dir` / повторное разрешение корня библиотеки в оркестраторе:
  в Фазе А путь пакета вообще не нужен; в Фазе Б путь инжектится с сервера
  (у него уже есть `WorldStore`), а не пере-выводится из env.
- Секция `"rag"` в топ-леве world.json: теряется при round-trip `WorldEnvelope`
  (см. п.3.4).
