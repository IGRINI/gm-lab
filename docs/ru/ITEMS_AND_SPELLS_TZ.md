# ТЗ: Предметы (описания + перенос сцена↔инвентарь) и заклинания (5e-подмножество)

Статус: Фаза И (предметы) → Фаза С (заклинания). Дизайн прошёл адверсариальную
панель; ключевые повороты и отвергнутые варианты зафиксированы в §5.

## 0. Зафиксированные факты (проверено по коду и панелью)

- Предметы НЕ являются канон-сущностями: тела живут только в `SceneState.items`;
  `Containment::Inventory` — мёртвый код; экшенов/валидатора/событий предметов нет.
- `view.rs:117` клонирует предметы ПРЕДЫДУЩЕЙ сцены в каждую пересборку вида —
  предметы «едут» за игроком при `move_player`, пока GM не перепишет их `set_scene`.
- Инвентарь ГГ — `Vec<String>`; путь загрузки — ручной `str_list`/`py_str`
  (объект → JSON-строка), путь применения — `as_str`-коэрция (объект → калека).
  Любой объектный формат записей требует атомарной миграции ~10 мест — НЕ в v1.
- `normalize_stat_value` НЕ рекурсит в объекты (`other => other`): вложенные
  `{current,max}` не получают числовой коэрции. hp работает потому, что ПЛОСКИЙ.
- Каталог тулов двухъярусный: `build_gm_tools()` байт-гейтнут фикстурой;
  `build_canon_gm_tools()` (move_player и пр.) НЕ в фикстуре — новые тулы туда
  стоят только правки `CANON_GM_TOOL_NAMES` + двух len-ассертов.
- Прецедент «движок не считает броски» (CHARACTERS_AND_STORY_TZ §К2) остаётся в
  силе: слоты/концентрация — СТЕЙТ (как hp), разрешение кастов — нотация roll_dice.
- Событие PLAYER_CHARACTER_UPDATE эмитит оркестратор (turn.rs), не world-метод.

## 1. Фаза И — предметы

### И1. Описания предметов: конвенция «название — описание» (БЕЗ смены типа)
- Записи `inventory`/`equipment` остаются строками. Конвенция: необязательный
  суффикс через « — » (длинное тире с пробелами): `"кинжал — 1d4, скрыт в сапоге"`.
- Голова (до первого « — ») = имя предмета; хвост = что он делает. Матчинг
  дельта-опов и тулов — ТОЛЬКО по голове (trim + to_lowercase, кириллица ок).
- Промпт: описание конвенции — в тул-описании update_player_character и одной
  строкой в GM_SYSTEM (карточка уже рендерит списки через «; »).
- UI (косметика, не блокер): DebugPanel/WorldDetailModal могут выделять голову
  жирным по сплиту « — »; ничего не ломается и без этого.
- `features` — без изменений (черты, не объекты).
- Типизированный ItemEntry {name, desc, charges…} — БУДУЩИЙ трек, гейт: реальная
  фича charges/attunement. До этого — не вводить (см. §5).

### И2. Пер-локационное хранение предметов сцены (фикс утечки — В СКОУПЕ)
- Новое поле `World.place_items: BTreeMap<String /*place_id*/, Vec<SceneItem>>`.
- При пересборке вида на СМЕНУ локации (`scene.location_id != anchor_id`):
  текущие `scene.items` стэшируются под старым location_id; новые items берутся
  из `place_items[anchor_id]` (нет — пустой список). Та же локация — как сейчас
  (`scene.items` живут). Путь `set_scene` НЕ меняется (он стейджит items для
  локации назначения ДО refresh — guard по location_id == anchor_id его сохраняет).
- Персист: аддитивный трейлинг-ключ `place_items` в world payload, эмит только
  когда непустой (старые сейвы байт-идентичны), парс с дефолтом. Roundtrip-тесты.
- Регресс-гейт: канон-тесты дополняются «предметы НЕ переезжают при move_player;
  предметы восстанавливаются при возврате в локацию; set_scene перекрывает стор».

### И3. Тулы take_item / drop_item (в build_canon_gm_tools, в конец)
- `take_item { item_id?, name?, reason? }`:
  1) `item_id` задан → точный матч по id (нет → tool_error unknown_item).
     Невидимые предметы берутся ТОЛЬКО по item_id (GM-доверенный путь).
  2) иначе по `name`: кандидаты = видимые предметы сцены,
     `name.trim().to_lowercase()`-сравнение. 0 → item_not_here (+ подсказка со
     списком видимых); >1 → ambiguous_item с кандидатами [{item_id,name,location}]
     — НИКОГДА не брать первый молча.
  3) `portable == false` → not_portable (описание тула учит: реджект — это
     фикшн, нарратив вместо ретрая; зеркало move_player-прозы).
  4) Успех: SceneItem удаляется из scene.items; в `pc.inventory` добавляется
     строка `"{name} — {details}"` (details пуст → просто name) через
     существующий changed-set (card_revision, дедуп по голове).
- `drop_item { name, location?, reason? }`:
  - матч по голове записи инвентаря (0 → tool_error); запись удаляется;
    в текущую сцену вставляется SceneItem {item_id: сгенерённый, name: голова,
    details: хвост, portable: true, visible: true, location: location|«рядом»}.
- Оба: эмит PLAYER_CHARACTER_UPDATE + SCENE_UPDATE из хендлеров оркестратора.
  КАНОН-СОБЫТИЙ НЕТ в этой фазе: приватные event-хелперы и валидатор — гейт
  канона; писать в append-only лог мимо них нельзя. Настоящие Action::TakeItem —
  в треке канон-тел предметов.
- GM_SYSTEM — чёткое правило выбора тула (тест, применимый моделью):
  «Предмет ЕСТЬ в CURRENT SCENE STATE → take_item (переносит тот же объект,
  сохраняя детали). Предмета в сцене НЕТ (подарок, крафт, покупка, найден в
  нарративе без стейджинга) → inventory_add. Никогда не inventory_add видимый
  предмет сцены; никогда не take_item того, чего в сцене нет. Выложить →
  drop_item; уничтожено/потрачено без следа в сцене → inventory_remove».
  Плюс кросс-ссылка в описании update_player_character.

## 2. Фаза С — заклинания (прагматичное 5e-подмножество)

### С1. Поля карточки (PLAYER_CHARACTER_FIELDS 26 → 29; все #[serde(default)],
безусловный эмит по 26-полевой дисциплине → re-bless голденов; старые сейвы
грузятся дефолтами; пакеты персонажей опаковы — дозаполнятся при следующем
save-back, это аддитивно)
- `spells: Vec<SpellEntry>`; SpellEntry { name: String, level: u8 (0 = заговор),
  concentration: bool, ritual: bool, effect: String }. Школа/дистанция/
  длительность/каст-тайм/апкаст — ПРОЗОЙ внутри effect (движок их не читает;
  панель вырезала 5 мёртвых полей).
- `spell_slots: Map<String,Value>` — ПЛОСКИЙ словарь «уровень → ОСТАТОК слотов»
  ({"1": 3, "2": 1}); попадает в dict_fields → К2-коэрция бесплатно.
- `spell_slots_max: Map<String,Value>` — авторские максимумы, тоже плоский,
  тоже dict_fields. Незаписанный уровень = слотов нет. Таблица 5e — только
  справка для авторов (в описании тула/доке), НЕ движковая механика.
- `concentration: String` — имя активного конц-заклинания, "" = нет. Текстовое
  поле, ДОКУМЕНТИРОВАННОЕ в схеме update_player_character (сброс концентрации
  без каста — явная правка поля; строка про это в GM_SYSTEM).
- Полная чек-листа мест роста (из панели): model.rs struct+Default;
  PLAYER_CHARACTER_FIELDS; классификация полей apply_player_character_fields
  (spells — новая категория «список объектов»: собственная коэрция через serde,
  НЕ as_str!); pc_field_value/set_pc_field; player_character_export;
  player_character_context; player_character_to_payload + from_payload (str_list
  НЕ трогает spells — отдельный парс); тул-схема; PlayerEditor (JSON textarea);
  WorldDetailModal/PlayerCard (read-only блок); re-bless: chat_payload голдены,
  gm_tools фикстуры, server state/debug голдены при необходимости.

### С2. Тул cast_spell (в build_canon_gm_tools, в конец)
- `cast_spell { name, slot_level?, reason? }`:
  1) спелл ищется в pc.spells по имени (trim+lowercase); нет → tool_error
     «персонаж не знает такого заклинания» (+ список известных в подсказке).
  2) level == 0 (заговор) → слот не тратится.
  3) иначе lvl = max(spell.level, slot_level|spell.level); spell_slots[lvl] > 0
     → декремент; иначе tool_error «нет свободных слотов уровня N» (описание
     тула: реджект — фикшн, нарратив сорванного каста, не ретрай).
  4) Концентрация: spell.concentration → pc.concentration = name; предыдущее
     значение возвращается как concentration_ended (GM нарративит обрыв).
  5) Результат: {spell, level, slot_spent_level|null, slots_remaining,
     concentration_started|null, concentration_ended|null} + card_revision bump
     + PLAYER_CHARACTER_UPDATE.
  6) НИКАКОЙ математики бросков: атака/спас/урон — существующий контракт
     roll_dice (нотация), апкаст-эффекты — проза effect. Ритуальный каст в v1 —
     нарративно (GM просто не вызывает cast_spell); арг as_ritual — не делаем.
- GM_SYSTEM: раздел про заклинания — каст ТОЛЬКО через cast_spell; слоты и
  концентрация движок-авторитетны; урон/спасы — roll_dice как раньше; сброс
  концентрации — поле concentration.

### С3. UI (v1 минимум)
- PlayerCard (DebugPanel) + WorldDetailModal: read-only блок «Заклинания»
  (имя, ур., конц-метка, effect) + строка слотов + активная концентрация.
- PlayerEditor: JSON textarea для spells/spell_slots/spell_slots_max (как
  abilities). Структурные редакторы и авторство в архитекторе персонажа — потом.
- NPC-спеллы — НЕ в этой фазе (прозу можно класть в knowledge/gm_notes).

## 3. Порядок и процесс
Фаза И (И1+И2+И3 один чанк: конвенция — это промпт/описания, основной код —
пер-локационный стор + два тула) → Фаза С (один чанк). Каждый чанк: опус-
имплементер по этому ТЗ → тесты/клиппи/байт-гейты → адверсариальные ревью →
чекпоинт-коммит.

## 4. Будущие треки (не делать сейчас)
- Канон-тела предметов: Item-сущности в WorldCanon + Action::TakeItem/DropItem/
  GiveItem через валидатор и engine::apply + канон-события + вид из
  Place.item_ids. Тогда же — give_item и инвентари NPC.
- Игрок как канон-актор (нужен is_player-гейт в 3 местах синтеза NPC).
- Типизированный ItemEntry (гейт: charges/attunement).
- Автоматика отдыха (восстановление слотов) — пока GM правит spell_slots руками.
- NPC-кастинг (innate-стиль «спелл → раз/день», по 5e-2024).
- Импорт SRD-компендиума заклинаний (CC-BY-4.0 атрибуция обязательна; механики
  свободны, Product Identity-имена — нельзя).

## 5. Отвергнутое панелью (чтобы не возвращаться)
- ItemEntry-структура в v1: ручной str_list/py_str-путь загрузки корёжит объекты
  в JSON-строки; as_str-коэрция apply-пути — второй независимый калечащий путь;
  UI рендерит [object Object]. Миграция ~10 мест атомарно — несоразмерна пользе,
  пока нет charges.
- Вложенный spell_slots {"1":{current,max}}: normalize_stat_value не рекурсит —
  строковые значения внутри никогда не коэрцируются, декремент ломается.
- Канон-события take/drop из тул-хендлера: private event()-хелперы и валидатор —
  обязательный гейт; сырой EventLog::append нарушает append-only инварианты.
- Дроп БЕЗ фикса утечки сцены: брошенный предмет «едет» за игроком — абсурд,
  порождаемый самой фичей (поэтому пер-локационный стор в скоупе И).
- as_ritual-ветка движка («вне боевого давления» — фикшн-суждение, движку нельзя).
- 10-полевая SpellEntry (school/casting_time/range/duration/upcast — мёртвые
  фри-стринги, движок не читает — в прозу effect).
- take/drop/cast в байт-гейтнутом build_gm_tools: канон-билдер дешевле (без
  re-bless), работает и в legacy-сценах (прецедент move_player).
